# Personal Guidelines

- Prefer `rg` or `rg --files` for text and file searches when available.
- Before broad `rg`/searches in large repos or generated-output-heavy trees, bound the result set in the command itself: first scope candidate directories or add recursive exclude globs such as `!**/node_modules/**`, `!**/target/**`, `!**/dist/**`, `!**/out/**`, `!**/build/**`, `!**/vendor/**`, plus repo-appropriate archive and generated-artifact excludes; prefer `rg --files` or `rg -l` to identify candidate files, then open only a small explicit selection with narrower globs. Treat output limits as display backstops, not search-scope controls.
- In repos likely to contain large untracked media, generated outputs, archives, or benchmark artifacts, do not start with full `git status --short`, `git status --ignored`, or `git ls-files --others`; first use `git status --short --untracked-files=no`, then inspect untracked files with counts, repo-appropriate excludes, or capped samples such as `git ls-files --others --exclude-standard | sed -n '1,40p'`.
- When using another checkout only as a protocol, API, or implementation reference, treat it as large-repo archaeology: start with `rg --files`, `rg -l`, `rg --count`, or exact service directories; exclude localization, generated assets, tests, build output, and dependency trees unless they are the target; only then run `rg -n` against one exact file or symbol window.
- When `.codex-tmp`, downloaded archives, or raw build/log artifact directories are part of the investigation, do not include the whole artifact tree in a broad source `rg -n`; first enumerate exact candidate files, then search selected files with `rg -l`, counts, snippets, or narrow `sed` windows.
- For embedded payloads or generated/minified long lines, such as base64 `data:` URLs in HTML/CSS or bundled assets, do not use raw `rg -n` if it would print the whole line; use `rg -l`, `rg -o` with a bounded pattern, counts, or structured length/snippet extraction.
