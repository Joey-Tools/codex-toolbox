# Personal Guidelines

- Prefer `rg` or `rg --files` for text and file searches when available.
- Before broad `rg`/searches in large repos or generated-output-heavy trees, bound the result set in the command itself: first scope candidate directories or add recursive exclude globs such as `!**/node_modules/**`, `!**/target/**`, `!**/dist/**`, `!**/out/**`, `!**/build/**`, `!**/vendor/**`, plus repo-appropriate archive and generated-artifact excludes; prefer scoped/capped probes such as `rg --count <pattern> <exact-dir> | head -n 80`, `rg --files <exact-dir> | head -n 80`, or `rg -l ... | head -n 80` to identify candidate files, then open only a small explicit selection with narrower globs. Treat output limits as display backstops, not search-scope controls.
- In repos likely to contain large untracked media, generated outputs, archives, or benchmark artifacts, do not start with full `git status --short`, `git status --ignored`, or `git ls-files --others`; first use `git status --short --untracked-files=no`, then inspect untracked files with counts, repo-appropriate excludes, or capped samples such as `git ls-files --others --exclude-standard | sed -n '1,40p'`.
- When using another checkout only as a protocol, API, or implementation reference, treat it as large-repo archaeology: for high-frequency terms such as common UI words, dates, or numerals, start with exact service directories, filename filters, or capped `rg --count` / `rg --files` / `rg -l` file lists before printing matches; exclude localization, generated assets, tests, build output, and dependency trees unless they are the target; only then run `rg -n` against one exact file or symbol window.
- When `.codex-tmp`, downloaded archives, or raw build/log artifact directories are part of the investigation, do not include the whole artifact tree in a broad source `rg -n`; first enumerate exact candidate files, then search selected files with `rg -l`, counts, snippets, or narrow `sed` windows.
- When inspecting GitHub Actions logs with `gh run view`, do not print full logs directly with `--log` or `--log-failed`; save them to a task-scoped file first, then use `gh run view --json ...` for metadata and `rg`, counts, tail, or short key-line excerpts on the saved log.
- When fetching public specs, standards, or web docs with `curl`, do not stream the full HTML or rendered text to the terminal; save it to a task-scoped file or request a bounded range/section, then extract headings, anchors, counts, or short snippets before opening only the relevant passage.
- For embedded payloads or generated/minified long lines, such as base64 `data:` URLs in HTML/CSS or bundled assets, do not use raw `rg -n` if it would print the whole line; use `rg -l`, `rg -o` with a bounded pattern, counts, or structured length/snippet extraction.
- For process diagnostics, avoid broad `ps` invocations that enumerate the full process table, such as `ps aux`, `ps -ef`, `ps -A`, `ps -e`, `ps axww`, or `ps -axo ...`; start with PID/name-scoped probes like `pgrep -af <pattern>`, `ps -p <pid> -o pid=,ppid=,stat=,etime=,command=`, or `lsof -nP -p <pid>`, and use counts or capped samples before printing broad process lists.
